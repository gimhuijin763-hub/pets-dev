# Product Context

## 문제 정의

### 현재 상황
많은 보호소와 입양센터는 입양 프로세스를 여러 도구에 분산하여 관리하고 있습니다:
- 동물 정보는 스프레드시트나 별도 문서로 관리
- 입양 신청은 이메일, 전화, 메신저 등 여러 채널로 접수
- 신청 상태 관리는 수기 또는 별도 시스템 사용
- 전체 현황 파악이 어렵고 관리 부담이 큼

### 핵심 문제
1. **정보 분산**: 입양 가능한 동물 정보를 한눈에 보기 어렵습니다
2. **채널 분산**: 신청 정보가 여러 채널로 들어와 확인이 번거롭습니다
3. **상태 관리 부재**: 접수 후 검토 상태를 일관되게 관리하기 어렵습니다
4. **현황 파악 어려움**: 운영자는 전체 신청 현황을 빠르게 파악하기 어렵습니다

### 근본 원인
문제는 프로세스가 복잡해서가 아니라, **핵심 흐름이 연결되어 있지 않기 때문**입니다.

## 사용자 가치

### 입양 희망자 관점
- **쉬운 탐색**: 입양 가능한 동물을 필터링하여 빠르게 찾을 수 있음
- **상세 정보**: 동물의 나이, 성별, 성격 등을 한눈에 확인
- **간편한 신청**: 복잡한 절차 없이 바로 신청서 제출
- **즉시 피드백**: 제출 완료 확인 및 안내 메시지

### 관리자 관점
- **통합 관리**: 모든 신청 내역을 한 곳에서 확인
- **빠른 처리**: 상태 변경을 드롭다운 한 번으로 즉시 처리
- **실시간 통계**: 접수/검토 중/승인/거절 건수를 한눈에 파악
- **효율적 운영**: 여러 도구를 오가지 않고 한 화면에서 모든 작업 처리

## 사용 시나리오

### 시나리오 1: 입양 희망자의 신청 흐름
1. 사용자가 랜딩 페이지에서 서비스 소개 확인
2. "데모 보기" 버튼 클릭하여 동물 목록 페이지로 이동
3. 필터(전체/개/고양이)를 사용하여 원하는 동물 탐색
4. 마음에 드는 동물 카드 클릭하여 상세 페이지 진입
5. 동물의 이름, 나이, 성별, 성격, 사진 확인
6. "입양 신청하기" 버튼 클릭
7. 신청서 폼에 이름, 연락처, 이메일, 신청 사유 입력
8. 유효성 검증 통과 후 "신청서 제출하기" 클릭
9. 제출 완료 화면에서 성공 메시지 확인
10. "다른 동물 보러 가기" 또는 상세 페이지로 복귀

### 시나리오 2: 관리자의 신청 관리 흐름
1. 헤더의 "관리자" 메뉴 클릭하여 관리자 페이지 진입
2. 상단 통계 카드에서 전체/접수/검토 중/승인/거절 건수 확인
3. 특정 상태 카드 클릭하여 해당 상태의 신청만 필터링
4. 신청 목록에서 동물 이미지, 신청자 정보, 신청 사유 확인
5. 상태 변경 드롭다운에서 새로운 상태 선택
6. 자동 저장 및 즉시 반영 확인
7. "새로고침" 버튼으로 최신 데이터 갱신

### 시나리오 3: 데모 발표 흐름
1. 랜딩 페이지에서 서비스 가치 제안 설명
2. 문제 정의 및 해결 방식 섹션 소개
3. 핵심 기능 카드 설명
4. "데모 보기" 클릭하여 실제 동작 시연
5. 동물 목록 → 상세 → 신청 → 제출 완료 흐름 시연
6. 관리자 페이지로 이동하여 방금 제출한 신청 확인
7. 상태 변경 시연 (접수 → 검토 중 → 승인)
8. 통계 카드 업데이트 확인

## 핵심 사용자 여정

### 입양 희망자 여정
```
랜딩 페이지 → 동물 목록 → 동물 상세 → 신청 폼 → 제출 완료
```

### 관리자 여정
```
관리자 페이지 → 신청 목록 확인 → 신청 상세 검토 → 상태 변경
```

## 차별화 포인트

### 기존 방식 vs ptes
| 구분 | 기존 방식 | ptes |
|------|----------|------|
| 동물 정보 관리 | 스프레드시트, 문서 | 웹 기반 통합 목록 |
| 신청 접수 | 이메일, 전화, 메신저 | 웹 폼 단일 채널 |
| 상태 관리 | 수기, 별도 시스템 | 드롭다운 즉시 변경 |
| 현황 파악 | 수동 집계 | 실시간 통계 대시보드 |
| 접근성 | 관리자만 | 누구나 조회 가능 |

## 제품 철학
- **최소 기능 집중**: 로그인, 결제, 채팅 없이 핵심 가치만 구현
- **빠른 데모 가능**: end-to-end 플로우 중심
- **운영 우선 설계**: 디자인보다 작동과 데모 안정성 우선
- **단순한 흐름**: 불필요한 단계 없이 가장 중요한 흐름만 설계
